# iOS Security Audit — 2026-05-30

**App:** Scrappy Kin iOS
**Codebase:** [github.com/Scrappy-Kin/scrappy-kin-ios](https://github.com/Scrappy-Kin/scrappy-kin-ios)
**Auditor:** AI security agent (Claude) under founder supervision
**Method:** Static analysis of open-source codebase. Not a penetration test.
**Scope:** OAuth/PKCE, keychain storage, Gmail send path, diagnostics, Capacitor bridge, StoreKit/subscriptions, build lanes, dependency manifest, audit tooling.

---

## Plain-language summary

Scrappy Kin sends removal request emails to data brokers from your Gmail account. The only Gmail permission it uses is `gmail.send` — it cannot read your inbox. Your personal information, Gmail authorization tokens, broker selections, and sent history stay on your device. There is no Scrappy Kin server that stores your data.

This audit checked whether those claims hold up in the code. They do. Four issues were found: three were fixed before launch, and one was accepted as an explicit tradeoff because adding remote control would contradict the no-server architecture.

## Findings

### Finding 1 — Gmail tokens could survive app deletion and reinstall
**Severity:** P1 — launch blocker
**Status:** Fixed before launch

**What the code showed:**
Gmail tokens are stored in the iOS keychain under the service name `cap_sec` via `capacitor-secure-storage-plugin` (`SecureStoragePlugin.swift:21-27`). iOS keychain data is not automatically removed when an app is deleted — this is documented behaviour. The plugin's own README shows the first-launch clear pattern (`README.md:185-198`). The app's `AppDelegate.swift` had only the default launch hook and did not clear the keychain on first install.

`app/src/services/secureStore.ts:173-177` provides an in-app wipe path, but that only runs while the app is installed.

**Why it mattered:**
A user connects Gmail, deletes the app believing their Gmail connection is gone, then reinstalls later. The old keychain entries are still there. If the refresh token is still valid, the new install could send Gmail messages as that prior user without re-authorisation.

This contradicts Scrappy Kin's device-local, user-in-control stance. Deleting the app should mean your Gmail authority is gone.

**What was fixed:**
`AppDelegate.swift` now clears the `cap_sec` keychain service on first launch after a fresh install. The sentinel is a `UserDefaults` key, which iOS does wipe on uninstall. Subsequent launches skip the clear.

### Finding 2 — No emergency path to disable Gmail sending in installed clients
**Severity:** P1 — launch blocker
**Status:** Policy accepted — documented below

**What the code showed:**
`app/src/services/gmailSend.ts:49-63` sends directly to Gmail after only a QA recipient guard and token lookup. There is no release-wide policy check, feature flag, or disable mechanism in the dispatch path (`app/src/services/sendQueue.ts:60-96`). The runtime dependency list includes no remote-config or emergency-message capability.

**Why it mattered:**
If a bug, bad broker list entry, or abuse report appeared after launch, the app has no way to warn users or pause sending without an App Store update. Users would need to update, uninstall, or manually revoke Google access.

**Decision:**
No remote kill switch will be added. A remote control mechanism would require the app to phone home, which contradicts the no-server, no-backchannel architecture.

The accepted emergency response path is:
1. Ship an App Store update as quickly as possible
2. Post instructions for users to revoke Scrappy Kin's Gmail access at [myaccount.google.com/permissions](https://myaccount.google.com/permissions) — this works immediately without an app update
3. Remove the app from the App Store to prevent new installs while a fix is in progress

This is an explicit architectural tradeoff. The window between a discovered bug and a shipped fix is bounded by App Store review time. That is accepted risk.

### Finding 3 — Keychain tokens were not marked non-migratable
**Severity:** P2
**Status:** Fixed before launch

**What the code showed:**
`capacitor-secure-storage-plugin` stored all keychain values with accessibility `.afterFirstUnlock` (`SecureStoragePlugin.swift:26`, `47`). This class allows keychain items to migrate across device restores and backups. Gmail tokens are the most sensitive secret in this app.

**Why it mattered:**
If a user restored a backup onto a new phone, the Gmail permission token could move with that backup. That is not the user expectation for this app: Gmail permission should belong to the device where the user connected it. This is weaker local custody than the architecture implies.

**What was fixed:**
Both write sites in `SecureStoragePlugin.swift` updated to `.afterFirstUnlockThisDeviceOnly`. Tokens are now non-migratable — they do not leave the device they were created on. Applied as a committed `patch-package` patch so it survives `npm install`.

---

### Finding 4 — Audit manifest script used the wrong working directory
**Severity:** P3 — tooling/auditability gap, not a runtime issue
**Status:** Fixed before launch

**What the code showed:**
`scripts/generate-audit-manifest.js:7` set `repoRoot = process.cwd()`. The npm script runs from `app/`, so the script looked for the manifest in the wrong place and failed with an error before doing any comparison.

**Why it mattered:**
The audit manifest is intended to make independent security review reproducible — it pins specific file versions so an auditor can fetch them at a known commit. A broken check means the gate doesn't fire. The app itself was not affected; the issue was that an audit reproducibility check could fail for the wrong reason.

**What was fixed:**
Line 7 changed to `path.resolve(__dirname, '..')`, so the script computes the repo root from its own location rather than wherever npm happens to be running from.

## Dependencies checked

Run from `app/`:

- `npm audit --audit-level=low` — passed, 0 vulnerabilities
- `npm ls --depth=0` — direct runtime dependencies are Capacitor, Ionic/React, routing, `ionicons`, and `capacitor-secure-storage-plugin`. No analytics, crash, or tracking SDK in direct dependencies.
- `npm test` — passed, 2 test files / 13 tests
- `npm run audit:manifest:check` — failed at audit time due to Finding 4 above; fixed before launch

Native dependencies use Swift Package Manager. `Package.resolved` pins `capacitor-swift-pm`, `ion-ios-filesystem`, and `SwiftKeychainWrapper`. No CocoaPods. The most security-relevant direct dependency is `capacitor-secure-storage-plugin` — it owns keychain behaviour for Gmail tokens. The accessibility class fix in Finding 3 was applied to this package.
