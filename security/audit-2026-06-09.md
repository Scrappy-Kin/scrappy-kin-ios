## Plain-language summary

Scrappy Kin sends data broker removal requests from the user's Gmail account. The app asks Google only for `gmail.send`, so it can send emails the user approves but cannot read the user's inbox.

On June 9, 2026, Scrappy Kin ran a second AI-assisted security review using Claude Fable 5 under founder supervision. The review found no launch-blocking issues and no P0/P1 findings. It re-confirmed the main trust lines: no inbox access, no Scrappy Kin account layer, no analytics or crash SDK, no hidden server custody, and no authenticated-account detection.

The review found four low-severity items worth recording. Three send-path hardening items were fixed before submission. One narrow Apple App Review test-recipient edge was left unchanged in code for launch and mitigated through visible in-app notice plus reviewer instructions to use test personal details.

## Findings

### Finding 1 - Hidden line breaks in email header fields

**Severity:** P3
**Status:** Fixed before submission
**Fix commit:** `c26a74c`

**What the code showed:**
The Gmail send code builds the MIME message that Gmail sends. Before the fix, the final MIME-building step did not reject hidden line breaks in the recipient, subject, or reply-to fields.

**Why it mattered:**
Email line breaks can change how an email header is interpreted. This was a low-risk edge because the app normally controls these values, but the final send step should still stop if a header field contains a hidden line break.

**What changed:**
`app/src/services/gmailSend.ts` now rejects `\r` or `\n` in `to`, `subject`, and `replyTo` before token lookup or network send. Regression tests were added for the guard.

---

### Finding 2 - Empty broker selection could send too broadly

**Severity:** P3
**Status:** Fixed before submission
**Fix commit:** `c26a74c`

**What the code showed:**
The send service had a fallback that could treat an empty broker selection as permission to use a broader broker list.

**Why it mattered:**
The app sends real emails to real broker addresses. If no brokers are selected, the safest behavior is to stop and ask for a selection, not infer a larger send target.

**What changed:**
`app/src/services/batchSend.ts` now fails closed when there is no explicit target broker list and no stored selected broker list. Onboarding still passes the starter broker IDs explicitly. The current user-facing message is: "No brokers selected. Please select at least one broker before sending."

---

### Finding 3 - Send code now re-checks profile fields before sending

**Severity:** P3
**Status:** Fixed before submission
**Fix commit:** `e887f63`

**What the code showed:**
The app already checked profile fields while the user filled them in. The send service did not fully re-run that same profile validation immediately before sending.

**Why it mattered:**
The send service is the last step before Gmail sends real email. It should not rely only on the earlier screen-level check, because saved local state can be older than the current screen.

**What changed:**
`app/src/services/sendQueue.ts` now calls the shared `getUserProfileValidationErrors` helper right before queue setup and send. This is not a second, competing validation rule set; the UI and send service use the same validation helper.

---

### Finding 4 - Apple App Review test-recipient path can receive reviewer-entered profile details

**Severity:** P3
**Status:** No code change for launch; mitigated and documented

**What the code showed:**
For Apple App Review, when the local profile email entered in the app is `app-review-redacted-02@example.invalid`, outgoing broker emails are routed to Scrappy Kin test inboxes instead of real broker addresses. This lets Apple test Gmail authentication, email generation, Gmail sending, and progress tracking without emailing real brokers.

**Why it mattered:**
If someone used the App Review profile email with real personal details, those details would be sent to Scrappy Kin test inboxes. The path is narrow and intended for Apple review, but it is still a real custody edge when that review profile is used.

**Why we did not change the trigger:**
The app does not detect which Gmail account is authenticated. Detecting the authenticated Google account would require identity/profile access or an account lookup, which would violate a more important privacy line: Scrappy Kin should not identify users just to route a review test path.

**Mitigation:**
The app shows an Apple App Review test-recipient notice before send when the review profile email is used. App Review instructions tell Apple to use test personal details with the provided review profile. This keeps the production Gmail send behavior close to the real app while avoiding unintended emails to real brokers.
